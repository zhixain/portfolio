🚀 簡介 (Elevator Pitch) 🚀 Elevator Pitch

打造一套無需網路、保護隱私，且能在低成本硬體上即時運作的 AI 人臉辨識系統。
我們結合深度學習與 FPGA 軟硬體整合技術，讓門禁可直接於本地端運算。 Building an AI facial recognition system that requires no network, protects privacy, and operates in real-time on low-cost hardware.
By integrating deep learning with FPGA HW/SW co-design, we enable access control to be processed entirely on the local edge device.
開發背景 Background

隱私與網路限制：高度依賴雲端運算，存在資料外洩風險，且無網路即停擺。

邊緣設備算力不足：若將 AI 模型直接放到一般平價的嵌入式設備上跑，往往會遭遇嚴重的畫面卡頓與高延遲。 Privacy & Network Limits: High dependence on cloud computing poses data leak risks and fails without a network.

Insufficient Edge Compute: Deploying AI models directly on standard budget embedded devices often leads to severe frame drops and high latency.
我們的解決方案 Our Solution

我們以 PYNQ-Z1 (FPGA) 作為邊緣加速平台，透過軟硬體協同設計 (HW/SW Co-design) 突破設備算力瓶頸：
• AI 模型硬體化：將複雜的 CNN 神經網路直接轉換為「實體硬體電路」執行，取代傳統軟體運算，大幅加速推論過程。
• ARM 與 PL 高效分工：結合 ARM 處理器與可程式邏輯 (PL)，精準分配控制邏輯與平行運算任務，打造完全脫離雲端的本地端架構。
• 兼顧隱私與即時效能：不僅實現資料完全在本地端 (Local) 處理以保障隱私，更在低成本硬體上成功實現 30 FPS 的即時辨識速度。 We use PYNQ-Z1 (FPGA) as an edge acceleration platform, breaking compute bottlenecks via HW/SW co-design:
• Hardware-accelerated AI: Converting complex CNNs directly into "physical hardware circuits" to replace traditional software computation, drastically speeding up inference.
• Efficient ARM & PL Division: Combining ARM with Programmable Logic (PL) to accurately distribute control logic and parallel computing tasks, creating a fully cloud-independent local architecture.
• Privacy & Real-time Performance: Ensuring data is processed entirely locally for privacy, while successfully achieving 30 FPS real-time recognition on low-cost hardware.

技術挑戰與效能優化 Technical Challenges & Optimization

🏆 模型維度取捨 (Trade-off) 🏆 Model Dimension Trade-off

  • 實作並比較了 512, 256, 128 維特徵輸出的模型。
  • 最終選定 128 維特徵模型，推論僅需 19.37 ms。
  • 相較於 512 維模型，取得 4.2 倍的推論加速，最適合記憶體受限的邊緣裝置。
  • Evaluated 512, 256, and 128-dim feature models.
  • Selected 128-dim model for its 19.37ms inference.
  • Achieved a 4.2x speedup compared to the 512-dim model, perfect for memory-constrained edge devices.

🧠 記憶體分配與底層傳輸 🧠 Memory & Low-level Transfer

  • 解決巨量 AI 參數在軟體與硬體間的傳輸延遲。
  • 透過 allocate 分配實體連續記憶體。
  • 利用 AXI Master 讓 FPGA 直接存取 DRAM，確保巨量權重與特徵圖的無縫高速傳輸。
  • Resolved transfer latency of massive AI parameters between SW and HW.
  • Used allocate for physically contiguous memory.
  • Enabled direct DRAM access via AXI Master, ensuring seamless high-speed transfer for weights and feature maps.
⚡ Vitis HLS 硬體加速與優化 ⚡ Vitis HLS Hardware Acceleration

我們導入關鍵的 HLS 優化指令以解決效能瓶頸：
1. 循環管線化 (Pipeline): 使用 #pragma HLS PIPELINE II=1 達成單一週期啟動，使迴圈運算高度並行化。
2. 陣列分割 (Array Partition): 將陣列分割至多個 Memory Bank，增加硬體存取埠 (Ports)，解決存取延遲。

We introduced critical HLS pragmas to solve performance bottlenecks:
1. Loop Pipelining: Used #pragma HLS PIPELINE II=1 for single-cycle initiation, highly parallelizing loop computations.
2. Array Partitioning: Split arrays across multiple Memory Banks, increasing hardware ports to resolve access latency.

操作流程與極端情境驗證 Workflow & Edge Cases Testing

  • 一鍵捕捉 (Button 0)：啟動 USB 攝影機並自動偵測人臉位置。
  • 離線註冊 (Button 1)：硬體加速提取 128 維特徵向量，並安全儲存於本地端 (SD Card)。
  • 即時解鎖 (Button 2)：拍攝新影像迅速計算餘弦相似度 (Cosine Similarity)，大於設定閾值即判定解鎖成功。
極端情境驗證 (Edge Cases Testing)：
經實際測試，系統在面對極端臉部條件（如露齒笑、閉眼、些微低頭）時，仍維持高度的特徵鑑別度與容錯率，能有效且精準地區分目標身分。
  • Capture (Button 0): Starts the USB camera and automatically detects face position.
  • Offline Register (Button 1): Hardware-accelerated 128-dim feature extraction, securely saved locally (SD Card).
  • Real-time Unlock (Button 2): Captures a new image, rapidly computes Cosine Similarity, and unlocks if greater than the threshold.
Edge Cases Testing:
Through practical testing, the system maintains high feature discrimination and fault tolerance even under extreme facial conditions (e.g., smiling with teeth, closed eyes, slight head tilt), effectively and accurately distinguishing target identities.

自我反思與未來展望 Reflection & Future Works

🔄 Overlay 動態重構優化 🔄 Dynamic Overlay Optimization

  • 現存瓶頸：系統需交替載入攝影機控制 (base.bit) 與模型推論 (design_1.bit) 的 Overlay，導致 FPGA 內部邏輯反覆重建，產生切換延遲。
  • 解決方案：未來考量將兩者整合成單一設計以消除重構成本；或導入雙 FPGA 架構，將影像擷取與模型推論分流，實現真正的即時並行處理。
  • Current Bottleneck: The system alternates loading camera control (base.bit) and inference (design_1.bit) overlays, causing repeated internal logic reconstruction and switching delays.
  • Solution: Plan to merge both into a single bitstream to eliminate reconfiguration cost, or introduce a dual-FPGA architecture to separate capture and inference for true real-time parallel processing.

📈 ASIC 設計流程轉換 📈 ASIC Flow Migration

  • 現存瓶頸：目前採用 FPGA Design Style，雖具備快速原型開發優勢，但通用邏輯資源在執行專用運算時，時脈與資源利用率易受限。
  • 解決方案：若面向大規模量產或追求更低延遲，計畫轉向 Standard Design Style (標準單元設計)，採用經優化的標準邏輯單元建構專用電路，藉此提升佈局密度與時脈效能。
  • Current Bottleneck: Currently utilizing FPGA Design Style. While advantageous for rapid prototyping, general logic resources often limit clock speed and resource utilization when executing dedicated computations.
  • Solution: For mass production or achieving lower latency, plan to migrate to Standard Design Style. Building dedicated circuits with optimized standard cells will enhance routing density and clock performance.
